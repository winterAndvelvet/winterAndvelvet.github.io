0x00 what is kubeadm?

kubeadm 通过执行必要的操作来启动和运行最小可用集群。提供了 kubeadm init 和 kubeadm join 的工具。

  • kubeadm init的任务：

    1. 环境检查-preflight。包括操作系统内核，Cgroup是否可用，端口是否被占用，docker是否安装等等。
    2. 生成证书，kube api的访问是tls,证书会存放在/etc/kubernetes/pki目录下。
    3. 生成其他组件访问apiserver的配置文件，路径是/etc/kubernetes,文件包括admin.conf, controller-manager.conf, kubelet.conf, scheduler.conf，其中admin是给kubectl操作时用的，告诉kubectl去连接哪个api server去操作。
    4. 会为api server, controller manager, scheduler, etcd生成pod配置文件，放在/etc/kubernetes/manifests路径下，kubelet会更具该路径下的配置文件，启动相应的pod。
    5. 此时master组件基本上启动了，kubeadm会检查 localhost:6443/healthz这个url，等待master组件完全启动。启动完成后，会为集群生成一个bootstrap token,后续节点拿到这个token后才能通过kubeadm join来加入这个集群。这个token会被保存在etcd中。
    6. 最后一步，安装默认插件kube-proxy和core-dns。他们是为整个集群提供服务发现和dns解析用的。这两个也是以pod的方式启动。

  • kubeadm join的任务：

    join的流程相比init来说简单不少，任何一台机器想要加入集群，必须要拿到server端的ca证书才能和其建立tls连接，那么如何在不手动拷贝的情况下拿到server的ca证书，这就需要指定一个token来证明node的合法性。于是便需要以下这样的命令来加入集群。加入后，ca.crt证书文件也将被存放到/etc/kubenetes/pki目录下。

      kubeadm join 192.168.0.233:6443 --token sd8a51.017go0q0mj7kiu4p \
  --discovery-token-ca-cert-hash sha256:94e0877bd0852665528c7fd124f6fdd8559283451e30747df3a35a3bc7dec1f2

0x01 环境准备

  • Centos8 2cpu 2g mini install
  • 安装脚本

    0x02 安装master

  1. sudo bash init_vm.sh

    环境ready,软件ready

  2. swap off and close firewall

    关闭系统交换区和防火墙

  3. sudo kubeadm init –config=kubeadm-config.yaml –upload-certs –pod-network-cidr=10.244.0.0/16

    使用kubeadm安装k8s环境，此操作成功后，k8s的master上的系统pod是ready(除了core-dns),其中–pod-network-cidr是中设置pod的网络范围，需要于kubeadm-conf里面保持一致。执行完之后需要根据输出提示将admin.conf 拷贝到$HOME/.kube/config下，使其当前用户可以使用kubectl命令

  4. 安装网络插件-flannel

    此操作成功后，k8s的master的组件应该ready

    0x03 安装node

  5. clone master机器

  6. 清理k8s环境，kubeadm reset后手动删除rm -rf $HOME/.kube/config

  7. kubeadm join xxxxx(根据master kubeadm init的输出)

    0x05 踩的坑和solution

  6. kube-flannel-ds crash,意思是flannel插件安装失败，故core-dns也没起来

    可能是与我之前kubeadm没有指定–pod-network-cidr有关，删除flannel，重新create。在master和node分别执行。

    然后清理使用kubedm reset清理环境，使用kubeadm init带–pod-network-cidr初始化后，再create flannel。

  7. 修改hostname和/etc/hosts 使其两台机器的dns解析没问题

  8. 在node节点运行kubectl出错：
    [root@localhost k8s-etp-practice]# kubectl get nodes
    The connection to the server localhost:8080 was refused - did you specify the right host or port?
    solution:

    出现这个问题的原因是kubectl命令需要使用kubernetes-admin来运行，解决方法如下，将主节点中的[/etc/kubernetes/admin.conf]文件拷贝到从节点相同目录下，然后配置环境变量： echo "export KUBECONFIG=/etc/kubernetes/admin.conf" >> ~/.bash_profile 立即生效 source ~/.bash_profile 接着再运行kubectl命令就OK了

0x06 总结

  1. master和node的pod和服务

    pod daemon mater node 功能
    apiserver yes no 提供了资源操作的唯一入口，并提供认证、授权、访问控制、API注册和发现等机制
    controller manager yes no 负责维护集群的状态，比如故障检测、自动扩展、滚动更新等
    scheduler yes no 负责资源的调度，按照预定的调度策略将Pod调度到相应的机器上
    kube-proxy yes yes 负责为Service提供cluster内部的服务发现和负载均衡
    add-on(kube-dns/Ingress Controller) yes no 负责为整个集群提供DNS服务/为服务提供外网入口
    Container runtime yes yes 负责镜像管理以及Pod和容器的真正运行（CRI）(docker/containerd)
    kubelet yes yes 负责维护容器的生命周期，同时也负责Volume（CVI）和网络（CNI）的管理
    etcd yes no 保存了整个集群的状态

  2. 分步确认k8s环境reday并排查问题

    • kubeadm init后：kube-scheduler、kube-proxy、kube-apiserver、kube-controller-manager这些pod应该是ready的。
    • 安装网络插件（flannel）后，coredns、kube-flannel也是ready的。
